# Message Media Attachments Plan

## Overview

Add N-per-message media attachments to BirGap, with end-to-end client-side encryption preserved. The current `MessageMedia` model is 1:1 with `Message` and has no HTTP endpoints wired up. This plan finalizes the feature as a Telegram-style album model: each message can carry up to 10 attachments (image / video / audio / document), each up to 100 MB, all client-encrypted.

**Design decisions:**

- **N attachments per message** (Telegram-style), not 1 (WhatsApp-style)
- **`MessageMedia.messageId` becomes nullable + non-unique**; new `userId` column tracks ownership for orphan cleanup and access control
- **Pre-staging allowed**: `MessageMedia` rows can be created in a `PENDING` state with `messageId = null`, then bound to a `Message` at send time — gives the mobile client resumable uploads
- **No drafts**: messages are still created at `POST /messages` time. Staging is for media only.
- **Album ordering**: by `createdAt` asc on the `MessageMedia` rows (no separate `position` column — keeps the schema minimal)
- **MIME allowlist per `MediaType`** enforced at presigned-URL time
- **Download is gated**: `GET /messages/media/:mediaId/download-url` calls `assertMediaAccess` (rewrite to take `mediaId`, join to `Message` for thread/group check)
- **Delete cascade**: on `FOR_EVERYONE` delete, enqueue `storage-cleanup` per `bucketKey` (and `thumbnailBucketKey` if present)
- **Orphan sweep**: new `media-cleanup` BullMQ queue, daily, deletes `PENDING` rows older than 24h and their R2 objects
- **Server can never decrypt** — `mediaCiphertextHash` and `thumbnailCiphertextHash` are opaque, server trusts client-asserted metadata

## 1. Database Schema Migration

**File:** `prisma/schema.prisma`

```prisma
model MessageMedia {
  id                      String       @id @default(uuid()) @db.Uuid
  userId                  String       @db.Uuid            // NEW: owner
  messageId               String?      @db.Uuid            // CHANGED: nullable, was @unique
  mediaType               MediaType
  bucketKey               String       @unique
  mimeType                String
  sizeBytes               Int
  filename                String
  thumbnailBucketKey      String?
  width                   Int?
  height                  Int?
  duration                Int?
  mediaCiphertextHash     String
  thumbnailCiphertextHash String?
  uploadStatus            UploadStatus @default(PENDING)
  uploadedAt              DateTime?
  uploadSessionId         String?      @unique @db.Uuid
  createdAt               DateTime     @default(now())

  user    User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  message Message? @relation(fields: [messageId], references: [id], onDelete: SetNull)

  @@index([userId, uploadStatus])
  @@index([messageId])
  @@index([bucketKey])
  @@index([uploadStatus, createdAt])
}
```

**Migration SQL** (auto-generated by `prisma migrate dev`):

- `ALTER TABLE "MessageMedia" ADD COLUMN "userId" UUID NOT NULL`
  - Backfill: in dev the table is empty, so `DELETE FROM "MessageMedia"` first, then add the column.
  - In prod: requires a backfill plan (out of scope for this week).
- `DROP CONSTRAINT "MessageMedia_messageId_key"` (drops `@unique`)
- `ALTER TABLE "MessageMedia" ALTER COLUMN "messageId" DROP NOT NULL`
- `CREATE INDEX "MessageMedia_userId_uploadStatus_idx" ON "MessageMedia"("userId", "uploadStatus")`
- `CREATE INDEX "MessageMedia_messageId_idx" ON "MessageMedia"("messageId")`
- `ADD FOREIGN KEY ("userId") REFERENCES "User"("id") ON DELETE CASCADE`
- `ALTER FOREIGN KEY "MessageMedia_messageId_fkey" ON DELETE SET NULL`

**Rationale for `onDelete: SetNull`:** If the parent message is hard-deleted, the `MessageMedia` row remains with `messageId = null` so the orphan sweep can clean up the R2 object. (Cascade would lose the bucket key.)

## 4. MediaService

**New:** `src/messages/media.service.ts`

Public methods:

- `initUpload(userId, dto) → { mediaId, uploadUrl, bucketKey }` — validates mime against the allowlist via `R2Service`, creates a `MessageMedia` row with `uploadStatus=PENDING`, `messageId=null`.
- `completeUpload(userId, mediaId, dto) → { mediaId, bucketKey, mediaType, mimeType, sizeBytes }` — asserts ownership (`media.userId === userId`), calls `r2.verifyObjectExists(bucketKey, sizeBytes)`, flips `uploadStatus=COMPLETE`, sets `uploadedAt=now`.
- `getDownloadUrl(userId, mediaId) → { downloadUrl, expiresIn }` — asserts the user can access the parent message (rewrite of `assertMediaAccess` to take `mediaId`), returns `r2.generateDownloadUrl(bucketKey)`.
- `assertAttachmentsOwned(userId, mediaIds) → MessageMedia[]` — internal, used by `MessagesService.send` and `GroupsService.queueGroupMessage`. Throws if any mediaId is missing, owned by another user, not `COMPLETE`, or already bound to a `messageId`.
- `cleanupMessageMedia(messageId)` — internal, used by `MessagesService.delete FOR_EVERYONE`. Fetches all `MessageMedia` rows for the message, enqueues a `storage-cleanup` job per `bucketKey` (and `thumbnailBucketKey`).

```typescript
if (dto.mediaIds?.length) {
  if (dto.mediaIds.length > maxAttachments) throw new BadRequestException(...);
  const media = await tx.messageMedia.findMany({
    where: { id: { in: dto.mediaIds }, userId: senderUserId, messageId: null, uploadStatus: 'COMPLETE' },
  });
  if (media.length !== new Set(dto.mediaIds).size) {
    throw new BadRequestException('Some mediaIds invalid, already attached, or not owned');
  }
  await tx.messageMedia.updateMany({
    where: { id: { in: dto.mediaIds } },
    data: { messageId: message.id },
  });
}
```

## 17. Risk & Notes

- **Production backfill**: existing `MessageMedia` rows lack `userId`. In dev the table is empty so the migration can `DELETE FROM "MessageMedia"`. In prod this needs a backfill plan (out of scope for this week).
- **MIME validation is client-trusted** — server doesn't sniff. A malicious client could upload an executable renamed to `.jpg`. Acceptable for E2EE (server can't decrypt to check anyway), but should be documented in the security model.
- **No `Range` support** for video scrubbing in clients — presigned GET doesn't support byte ranges with R2's S3 API in the same way. Document for mobile team.
- **Album ordering**: by `createdAt` asc, no separate `position` column. If clients need custom order, add a `position Int` column later.
- **Group fanout size**: 10 × 100MB = 1GB stored per group message. Acceptable, but consider per-group media quota (out of scope this week).
